# Graph Store

The `graphStore` is the single source of truth for the active Redstring session. It is implemented using **Zustand**, a lightweight state management library.

## Architecture: The Monolithic Store

In the current alpha architecture, we made a deliberate choice to use a **Monolithic Store Pattern**. Instead of splitting state into many small slices, the entire graph state (nodes, edges, selection, view transform) lives in one massive store.

> [!IMPORTANT]
> **Why Monolith?**
>
> 1.  **Transactional Integrity**: Complex operations (like "Drag 50 nodes and update their connected edges") happen in a single render cycle.
> 2.  **Serialization**: The entire state can be `JSON.stringify`'d instantly for saving or undo/redo history.
> 3.  **Simplicity**: New features can be added without creating "state synchronization hell" between multiple stores.

### Key Slices

While monolithic, the store is internally organized into logical "slices" of functionality:

1.  **Graph Data**: `nodes` (Map), `edges` (Map).
2.  **Selection State**: `selectedNodeIds` (Set).
3.  **Viewport**: `transform` { x, y, k (zoom) }.
4.  **Interaction State**: `connectingNodeId`, `dragging`.

## Usage in Components

Components consume the store via hooks. Use selective subscriptions to prevent unnecessary re-renders!

**✅ Good (Selective)**:
```javascript
// Only re-renders when node count changes
const nodeCount = useGraphStore(state => state.nodes.size);
```

**❌ Bad (Full Subscription)**:
```javascript
// Re-renders on ANY change (mouse move, selection, etc.)
const state = useGraphStore();
```

## Performance Optimization

Because the store is large, we use **Immer** middleware to allow mutable syntax while keeping state immutable.

```javascript
set(produce(state => {
  state.nodes.get(id).position.x += 5; // Mutable syntax
  // Immer handles the immutable update under the hood
}))
```

## Future Roadmap

As the application scales beyond 10,000 nodes, we plan to "shuck" this monolith into a **Spatial Database** model (using `sqlite-wasm` or similar), treating the Zustand store only as a "viewport cache" for visible items.
